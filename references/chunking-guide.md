# 切块策略指南

## 为什么切块很重要

RAG 系统的效果很大程度上取决于**切块质量**。块太大则中心内容被稀释，块太小则丢失上下文。kb-builder 提供了三种切块策略，适用于不同的文档类型。

---

## 三种策略

### 策略 A：结构感知切块（Structure-aware）

**适用场景：** 带 Markdown 标题结构的长文章（如技术博客、教程、产品文档）

**行为：**
- 根据 `##`（二级标题）和 `###`（三级标题）切分
- 每个块最多 800 个字符（超过则从段落边界截断）
- 块与块之间保持标题层级关系

**示例：**

```markdown
## 安装指南
内容...
### macOS
内容...    → 切为一块
### Windows
内容...    → 切为一块
## 配置说明
内容...    → 切为一块
```

**优点：** 每个块都自带标题上下文，检索时更容易定位
**缺点：** 如果标题结构混乱，切出的块可能不均匀

### 策略 B：模板/段落切块（Template/Paragraph）

**适用场景：** 简短说明、简报、没有分层结构的内容

**行为：**
- 按空行（段落边界）切分
- 每个块最多 600 个字符
- 如果单段超过 600 字符，在句子边界截断

**优点：** 实现简单，适合无结构文本
**缺点：** 不支持跨段上下文关联

### 策略 C：问答对切块（Q&A Pair）

**适用场景：** 社区问答、论坛讨论、FAQ（当前为预留策略，尚未在索引器中实现）

**行为：**
- 将社区数据解析为 `(问题, 回答)` 对
- 将问题和回答拼接成一个块
- 后续支持检索时只匹配问题，返回问题和回答

**优点：** 检索精确度最高（问题匹配问题）
**缺点：** 依赖数据格式标准化，未在 `index.py` 中实现

---

## 如何选择策略

| 文档类型 | 推荐策略 | 原因 |
|---------|---------|------|
| 技术文章（带 ##/### 标题） | A - 结构感知 | 标题就是天然的切分点 |
| 简短简报（500 字以内） | B - 模板/段落 | 结构简单，段落即可 |
| 混杂内容 | A - 结构感知 | 有标题用标题，无标题用段落 |
| GitHub Issues / 论坛帖 | C - 问答对 | 原生 Q&A 格式（未来支持） |

> **经验法则：** 如果你的文档有 `##` 标题，选 A。没有标题或标题随意，选 B。

## FAQ

### 块太大怎么办？

**场景：** 检索结果包含大量无关信息，回答泛泛。
**原因：** 块包含的内容过多，核心信息被稀释。

**解决：**
- 缩小最大字符数（编辑 `config.yaml` 的 `chunk_size`）
- 检查文档是否缺少标题——添加 `##` 标题让切分更精细
- 对于策略 B，减小 `max_chunk_size`（默认为 600）

### 块太小怎么办？

**场景：** 检索到的块上下文不完整，回答断章取义。
**原因：** 块只包含了一小段，缺少前后文。

**解决：**
- 增大 `chunk_overlap`（块重叠字符数），让相邻块共享上下文
- 或者切换为策略 A，利用标题聚合更自然的块
- 如果文档本身就是短内容（如 200 字说明），策略 B 直接以小段落处理即可

### 什么时候应该换策略？

- 你发现检索结果经常来自错误的部分 → 用策略 A 试试
- 你的文档从长文章改为短简报 → 切换到策略 B
- 你开始索引社区问答数据 → 期待策略 C 实现

### 如何配置策略？

编辑 `config.yaml`：

```yaml
chunk_strategy: "A"       # A | B | C
chunk_size: 800           # 最大字符数
chunk_overlap: 100        # 块之间的重叠字符数
```

修改后需重新运行索引：

```bash
source .venv/bin/activate
python index.py --full
```
