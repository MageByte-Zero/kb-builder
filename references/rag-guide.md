# RAG 基础指南

## 什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种让大语言模型（LLM）能够"查阅资料"后再回答问题的技术方案。它的核心思想是：**不依赖模型自身的记忆，而是先去知识库中检索相关内容，再把检索结果作为上下文提供给模型生成答案**。

RAG 解决了 LLM 的两个天然缺陷：

1. **知识截止日期** — 模型训练完成后，新产生的知识它并不知道
2. **幻觉问题** — 模型可能编造它不确定的事实

通过 RAG，你可以让 LLM 基于你提供的文档（技术博客、产品文档、历史文章等）给出有依据的回答。

---

## 三步流程

kb-builder 的 RAG 流程分为三个阶段：

### 1. 索引（Indexing）

将原始文档切分为小块（chunk），对每个块计算向量（embedding），存入向量数据库。

```
原始文档 → 切块 → 向量化 → ChromaDB
```

这是"建库"阶段，只需执行一次（或文档更新时重新执行）。

### 2. 检索（Retrieval）

收到用户问题时，将问题也转换为向量，在向量数据库中搜索最相似的文档块。

```
用户问题 → 向量化 → 向量相似度搜索 → 返回 top-k 最相关块
```

### 3. 增强生成（Augmented Generation）

将检索到的文档块作为上下文，连同用户问题一起发给 LLM，让 LLM 基于上下文回答。

```
[用户问题] + [检索到的文档块] → LLM → 有依据的回答
```

## 为什么不用关键词搜索

关键词搜索（如 Elasticsearch、grep）基于**精确匹配**：文档必须包含用户输入的词语才能被找到。这在以下场景中会失效：

| 场景 | 关键词搜索 | RAG 向量搜索 |
|------|-----------|-------------|
| 用户搜"如何部署"但文档写的是"安装步骤" | 搜不到 | 能匹配（语义相近） |
| 用户搜"数据库优化"但文档写的是"慢查询排查" | 搜不到 | 能匹配 |
| 同义词："修复"vs"解决"、"创建"vs"新建" | 搜不到 | 能匹配 |
| 英文术语 vs 中文说法："OOM"vs"内存溢出" | 搜不到 | 能匹配 |

向量搜索基于**语义相似度**，即使措辞不同，只要意思相近就能检索到。这在知识库内容多、用户提问方式不可控的情况下至关重要。

## 为什么不全量发给 LLM

有人会问：既然 LLM 上下文窗口越来越大（128K、200K），为什么不直接把所有文档发给 LLM？

主要有两个问题：

### 成本问题

LLM 按 token 计费（或按上下文长度消耗本地资源）。假设你的文档有 10 万 token：

- 每次提问都发 10 万 token，成本是只发问题的几十倍
- 如果每天提问 100 次，费用差距巨大
- 本地模型虽然不花 API 费，但推理速度会显著下降

### 质量衰减（Lost in the Middle）

大量研究（Liu et al., 2024 等）表明：**LLM 对长上下文中部内容的关注度显著低于开头和结尾**。当上下文过长时：

- 模型容易忽略关键信息
- 检索到的相关内容被无关内容"淹没"
- 回答质量反而下降

RAG 的精髓就是：**只给 LLM 它需要的那一小部分，而不是所有内容**。这既节省了成本，又提高了回答质量。

---

## 什么时候需要 RAG

- 你的知识库超过几百条内容
- 知识库会持续更新
- 用户提问方式是自然语言（而非关键词）
- 需要准确引用来源

如果只是几个固定模板或静态 FAQ，用简单的关键词搜索或规则匹配就够了，不需要 RAG。
